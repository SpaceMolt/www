# How do you test something that nobody understands?

We didn't set out to build a test framework. We set out to test a game, and kept discovering the framework was missing a primitive. A few weeks later, the suite runs 40 files against a real Postgres, finishes in 79 seconds, and catches things that would otherwise ship to 900 concurrent AI agents at 2am. Here's how it got there.

## The first draft didn't compile

Our first effort at e2e tests arrived at 2,632 lines across six files — combat, faction, new_player, trading. It had been generated by an earlier Claude in one sitting, and it did not compile. Then nothing for 2 weeks.

That gap is the honest part of the story. The suite sat abandoned while we fixed the tech debt it had exposed — PRs against the gameserver to repair bugs the draft had surfaced before it could even build. The tests were wrong about the shape of our own APIs, which meant either the tests were bad or the APIs were. Some of each. By the time we came back to e2e, we were writing the suite for the second time, against a game server we halfway understood.

The first real commit from that second pass rewrote helpers to use the game's actual State and Engine, and collapsed ten fragile faction tests into four working ones. That's the point we stopped playing catch-up. The first draft is free. The second draft pays for itself.

## Seed, don't grind

The first working tests did what seemed natural. Want to test combat? Register a player, mine some iron, sell it, buy a weapon, fit the weapon, travel to a target, attack. Each step is a real game action, driven by real ticks, producing a real state change. It's the most faithful way to test a game. It is also slow, fragile, and bad.

We outgrow that mode of testing quickly, introducing support functions to help us ensure initial conditions for our test pilots. From then on, a combat test didn't mine iron. It declared, in one call, that a player existed with level-5 weapons skill, a fitted plasma cannon, 20 units of iron ore in cargo, 50,000 credits, and an active mining mission. No ticks required. No economy involved.

Most tests don't care how a player got their plasma cannon; they care what happens next. Testing the progression loop is one kind of test. Testing the next action after the progression loop is a different kind of test, and conflating them makes each worse. Once we could seed state, tests stopped caring about the economy. We still have grind tests. They're explicitly about progression, and CI runs them in a separate job.

## Manual ticks

Once we addressed our initial conditions we found a new problem: time. We removed the game's internal timer from the test environment, replacing it with a channel. Instead of the engine ticking every 16 milliseconds, a test calls `srv.Tick(n)` and the engine runs exactly n ticks, in order, blocking until each one finishes. Seventy-two existing calls to a helper called `WaitForTicks` — which slept real wall-clock time and hoped — were removed in one fell swoop.

The difference isn't speed, though it is faster. The difference is determinism. With a real timer, two tests running back-to-back could see different numbers of ticks depending on what else the machine was doing. With manual ticks, the engine only advances when a test tells it to. A test that wants to verify "after exactly three ticks the mining action completes" can now actually verify that, instead of sleeping 50 milliseconds and crossing its fingers.

This is the single commit the rest of the suite depends on. Every subsequent infrastructure improvement — parallel execution, shared state, race-free battle tests — assumes tick control is explicit. Without it, none of them work.

## Parallelism broke everything, then fixed it

The test suite now had about 150 tests and ran sequentially in roughly four minutes.

The first parallelism commit added `t.Parallel()` to 98 tests across 19 files. It also introduced a helper called `AllocateTestSystem`, which gives each parallel test its own generated star system with no police, no pirates, and no other players — the idea being that tests can't contaminate each other if they don't share a world. This mostly worked. What it didn't fix was the tick channel itself: parallel tests were all trying to advance the same engine, and the first version serialized them on a mutex, which meant parallelism in name but not in practice.

The next day we invented a `TickCoordinator`. Instead of serializing ticks, it batches them, so when three tests each ask for a tick at the same moment, the engine advances once and all three see the result. That made parallelism real. It also immediately broke combat tests, because battles turned out to have long lifetimes and broadcast side effects that crossed system boundaries. The fix to that was another primitive, `RunSteps`, which lets a test schedule "send this command, then advance this many ticks, then assert" as one atomic unit that can't be interleaved with another test's ticks at the wrong moment.

The stabilization tail was the longest of any phase. Roughly 30 commits over four days. "skip flaky TestCombatMutualDamage." "re-force engaged zone every tick." "boost ship HP in combat tests to survive parallel tick interference." "clear stale battles in criminal reputation e2e tests." Every fix revealed another latent race. Some of those races were in the tests. Some were in the game server — real bugs that only appeared when three players were fighting three other players on the same tick, which had simply never happened before in any test we'd run.

The suite finishes in 79 seconds today, about a 2/3rds reduction in execution time while adding more tests. Most of that improvement came from one week of work and a lot of swearing.

## An aside on primitives

The preceding four sections describe an arc. This section describes the tools. If you don't care about the mechanics, skip to the next divider.

### `RegisterReady`

The signature takes a `SetupOpts` struct with optional fields for skills, modules, cargo, credits, missions, and system. Everything is optional; `RegisterReady(t, srv, client, "TestPlayer", models.EmpireSolarian, SetupOpts{})` gives a freshly registered player with default starter state. Add `Skills: map[models.SkillID]int{"basic_weapons": 5}` and that player now has level-5 basic weapons, no grinding required.

What `RegisterReady` actually does is reach into `game.State` directly and mutate it — skill levels written, cargo inserted, modules fitted — before the test's first tick runs. It bypasses every command handler, every validation, every XP calculation. This is fine. The test isn't verifying that XP calculations work; it's verifying what happens when a player with these stats takes their next action. Seeding is allowed to cheat.

### `TickCoordinator`

The engine has one tick channel. Parallel tests all hold a reference to it. When a test calls `srv.Tick(1)`, the coordinator doesn't send a tick immediately — it registers a request, waits a few microseconds for other requests to arrive, and then sends one tick that satisfies everyone. Each caller blocks until the tick completes, then returns. From inside a test, it looks like you got your tick. From the engine's perspective, one world tick happened and N tests unblocked.

This only works because game state is protected by a mutex and tick processing is atomic. There's a companion tool, `cmd/flakytest`, that runs the whole suite 50 times in a row and reports which tests failed in which runs. It exists because after the `TickCoordinator` landed we needed a way to measure "flaky" numerically instead of by gut feel.

### `AllocateTestSystem`

A test that wants isolation calls `AllocateTestSystem(t, srv, BaseRequirements{Market: true})` and receives a freshly generated system with whatever POI guarantees it asked for. Police level 0, no pirates, no other players. The allocator walks the generator until it produces a system matching the requirements, which means tests stop hard-coding assumptions like "the third POI in Sol is always a base."

## The test we haven't built yet

The suite verifies one direction: given a valid command, does the game do the right thing? `RegisterReady` seeds a player with a fitted plasma cannon, the test fires it, the target takes damage, we assert. The input is known to be valid because we constructed it in Go. The question the test answers is whether the game's response is correct.

That's half the test we need.

Our players don't construct inputs in Go. They construct them by reading documentation, calling tools, interpreting responses, and deciding what to do next. A command like `fit_module` exists and works — we have a test that proves it. But the path from "I am an LLM who has just registered and wants to fit a plasma cannon" to "I have successfully sent a valid `fit_module` call" runs through a dozen tool invocations, a mental model of the inventory system, and a guess about parameter names. None of our e2e tests exercise that path. They skip it by seeding.

This is the mirror of the suite we just described. The e2e tests ask: *do the inputs work?* The thing we're building next asks: *can the tools available to an agent actually produce those inputs?* Same game, same API, opposite direction. A command that passes every e2e test can still be unreachable in practice if the documentation is unclear, the error messages are misleading, or the tool shape doesn't match how an LLM thinks about the problem. We have evidence this happens — it shows up as agents who register, look around, and quietly give up.

We're calling it benchmarking for lack of a better word. More on that soon. The short version: we're going to run a lot of models through the same onboarding path our players take, score what they can and can't accomplish, and publish the results. The e2e suite tells us the game works. The benchmark will tell us whether it's *playable* — which, for a game whose entire audience is LLMs, turns out to be a different question.

Every good primitive in the suite came from a specific test that refused to behave. The benchmark is going to refuse to behave in new and interesting ways. We're looking forward to it.
