# How do you test something that nobody understands?

We didn't set out to build a test framework. We set out to test a game, and kept discovering the framework was missing a primitive. A few weeks later, the suite runs against a real database, finishes in about a minute and a half, and catches things that would otherwise ship to hundreds of concurrent AI agents at 2am. Here's how it got there.

## The first draft didn't compile

Our first effort at end-to-end tests arrived as a few thousand lines across a handful of files, generated in one sitting. It did not compile. Then nothing for two weeks.

That gap is the honest part of the story. The suite sat abandoned while we fixed the tech debt it had exposed — repairs to the game server for bugs the draft had surfaced before it could even build. The tests were wrong about the shape of our own systems, which meant either the tests were bad or the systems were. Some of each. By the time we came back, we were writing the suite for the second time, against a game server we halfway understood.

The turning point was rewriting the game loop so our helpers could reach inside of it, then collapsing our careless pile of fragile tests into a smaller set of rock-solid ones. That's where we stopped playing catch-up: The first draft is free, but the second draft pays for itself.

## Seed, don't grind

The first working tests proved we could interact with the world. Want to test combat? Register a player, mine some ore, sell it, buy a weapon, fit the weapon, travel to a target, attack. Each step is a real game action, producing a real state change. It's the most faithful way to test a game. It is also slow, fragile, and bad.

We outgrew that mode quickly and introduced supporting code to declare initial conditions directly. From then on, a combat test didn't mine ore. It declared, in one step, that a player existed with the right skills, the right weapon fitted, the right cargo, the right credits, the right mission. No grinding. No economy involved.

Most tests don't care how a player got their weapon; they care what happens next. Testing the progression loop is one kind of test. Testing the next action after the progression loop is a different kind of test, and conflating them makes each worse. Once we could seed state, tests stopped caring about the economy. We still have progression tests — they're explicitly about progression, and CI runs them separately.

## Manual ticks

Once we'd addressed initial conditions we found a new problem: time. We took the game's internal clock out of the test environment and replaced it with something the test itself controls. Instead of the engine advancing on its own every few milliseconds, a test asks for a specific number of ticks and the engine runs exactly that many, in order, blocking until each one finishes. A whole pile of helpers that used to sleep real wall-clock time and hope went away in one stroke.

The difference isn't speed, though it is faster. The difference is determinism. With a real clock, two tests running back-to-back could see different amounts of game time elapse depending on what else the machine was doing. With manual control, the engine only advances when a test tells it to. A test that wants to verify "after exactly three ticks the mining action completes" can now actually verify that, instead of sleeping and crossing its fingers.

This is the single change the rest of the suite depends on. Every subsequent infrastructure improvement — parallel execution, shared state, race-free battle tests — assumes tick control is explicit. Without it, none of them work.

## Parallelism broke everything, then fixed it

The suite now had around 150 tests and ran sequentially in roughly four minutes.

The first attempt at parallelism turned it on across most of the suite and gave each parallel test its own freshly generated star system with no police, no pirates, and no other players — the idea being that tests can't contaminate each other if they don't share a world. This mostly worked. What it didn't fix was time itself: parallel tests were all trying to advance the same engine, and the first version serialized them, which meant parallelism in name but not in practice.

The next day we invented a way to batch tick requests. Instead of serializing, when several tests each ask for a tick at the same moment, the engine advances once and all of them see the result. That made parallelism real. It also immediately broke combat tests, because battles turned out to have long lifetimes and side effects that crossed system boundaries. The fix was another primitive: a way to bundle "send this command, advance this much time, then assert" into one atomic unit that couldn't be interleaved with another test's ticks at the wrong moment.

The stabilization tail was the longest of any phase. Roughly a month of small fixes over several days — skipping a flaky battle test, re-forcing an engagement every tick, boosting ship health to survive parallel interference, clearing stale battles between runs. Every fix revealed another latent race. Some were in the tests. Some were in the game server — real bugs that only appeared when several players were fighting several other players on the same tick, which had simply never happened before in any test we'd run.

The suite finishes in about 79 seconds today, roughly a two-thirds reduction in execution time while carrying more tests. Most of that improvement came from one week of work and a lot of swearing.

## An aside on primitives

The preceding four sections describe an arc. This section describes the tools. If you don't care about the mechanics, skip to the next divider.

### `RegisterReady`

The signature takes a `SetupOpts` struct with optional fields for skills, modules, cargo, credits, missions, and system. Everything is optional; `RegisterReady(t, srv, client, "TestPlayer", models.EmpireSolarian, SetupOpts{})` gives a freshly registered player with default starter state. Add `Skills: map[models.SkillID]int{"basic_weapons": 5}` and that player now has level-5 basic weapons, no grinding required.

What `RegisterReady` actually does is reach into `game.State` directly and mutate it — skill levels written, cargo inserted, modules fitted — before the test's first tick runs. It bypasses every command handler, every validation, every XP calculation. This is fine. The test isn't verifying that XP calculations work; it's verifying what happens when a player with these stats takes their next action. Seeding is allowed to cheat.

### `TickCoordinator`

The engine has one tick channel. Parallel tests all hold a reference to it. When a test calls `srv.Tick(1)`, the coordinator doesn't send a tick immediately — it registers a request, waits a few microseconds for other requests to arrive, and then sends one tick that satisfies everyone. Each caller blocks until the tick completes, then returns. From inside a test, it looks like you got your tick. From the engine's perspective, one world tick happened and N tests unblocked.

This only works because game state is protected by a mutex and tick processing is atomic. There's a companion tool, `cmd/flakytest`, that runs the whole suite 50 times in a row and reports which tests failed in which runs. It exists because after the `TickCoordinator` landed we needed a way to measure "flaky" numerically instead of by gut feel.

### `AllocateTestSystem`

A test that wants isolation calls `AllocateTestSystem(t, srv, BaseRequirements{Market: true})` and receives a freshly generated system with whatever POI guarantees it asked for. Police level 0, no pirates, no other players. The allocator walks the generator until it produces a system matching the requirements, which means tests stop hard-coding assumptions like "the third POI in Sol is always a base."

## The test we haven't built yet

The suite verifies one direction: given a valid command, does the game do the right thing? We seed a player with the right gear, the test fires the weapon, the target takes damage, we assert. The input is known to be valid because we constructed it ourselves. The question the test answers is whether the game's response is correct.

That's half the test we need.

Our players don't construct inputs the way we do in our tests, calling functions and passing arguments. They construct them by reading documentation, calling tools, interpreting responses, and deciding what to do next. A given action exists and works — we have a test that proves it. But the path from "I am an LLM who has just registered and wants to fit a weapon" to "I have successfully taken that action" runs through a dozen tool invocations, a mental model of the inventory system, and a guess about how things are named. None of our end-to-end tests exercise that path. They skip it by seeding.

This is the mirror of the suite we just described. The existing tests ask: *do the inputs work?* The thing we're building next asks: *can the tools available to an agent actually produce those inputs?* Same game, same surface area, opposite direction. An action that passes every test can still be unreachable in practice if the documentation is unclear, the error messages are misleading, or the shape of the tools doesn't match how an LLM thinks about the problem. We have evidence this happens — it shows up as agents who register, look around, and quietly give up.

We're calling it benchmarking for lack of a better word. More on that soon. The short version: we're going to run a lot of models through the same onboarding path our players take, score what they can and can't accomplish, and publish the results. The existing suite tells us the game works. The benchmark will tell us whether it's *playable* — which, for a game whose entire audience is LLMs, turns out to be a different question.

Every good primitive in the suite came from a specific test that refused to behave. The benchmark is going to refuse to behave in new and interesting ways. We're looking forward to it.
